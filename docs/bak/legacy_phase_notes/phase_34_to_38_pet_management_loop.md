# Phase34-38：宠物管理闭环

## 目标

这一组阶段把 Phase33 的“捕捉后生成宠物实例”接成战斗外可操作的宠物管理系统。

第一版仍然只做 Godot 本地逻辑，不接 Node.js、不接 MySQL、不做正式美术资源。

## 阶段范围

### Phase34：宠物栏 UI

- 地图界面提供 `宠物` 入口。
- 打开后显示玩家当前宠物。
- 同一个面板同时服务 PC 和手机；PC 只是用鼠标点击，不做单独复杂布局。
- 宠物列表显示：名称、等级、状态、生命、经验。

### Phase35：出战 / 待机 / 休息状态管理

- 玩家通过一个状态按钮循环 `休息 -> 战斗 -> 待机 -> 休息`。
- 玩家最多只能有一只出战宠，也允许没有出战宠。
- 主宠状态为 `battle`。
- 可出战宠状态为 `standby`。
- 休息宠状态为 `rest`，当前不会参加战斗。
- 仓库宠状态为 `storage`，不能直接设置为主宠。

### Phase36：兽栏 / 仓库雏形

- 队伍上限为 5 只。
- 非仓库宠物属于随身队伍。
- 队伍已满时捕捉的新宠物进入 `storage`。
- 第一版用同一个宠物面板查看队伍和兽栏，不单独做大地图建筑。

### Phase37：宠物详情页

详情区显示：

- 种系、亚种、形态。
- 属性配比。
- 主动技能列表。
- 被动技能说明。
- 攻击、防御、敏捷、生命。

### Phase38：存档和自动验证

- 宠物操作会写回 `user://player_profile.json`。
- 自动测试覆盖：
  - 打开宠物面板。
  - 选中宠物能看到详情。
  - 状态按钮按 `休息 -> 战斗 -> 待机 -> 休息` 循环。
  - 休息宠点一次变战斗，点两次变待机。
  - 出战宠变待机后不会自动让其他宠物接替出战。
  - 仓库宠不可直接设为主宠。
  - 队伍满时捕捉宠物进入仓库。
  - 下一场战斗读取新的主宠。

## 数据规则

宠物实例继续使用 Phase33 字段：

- `instanceId`
- `formId`
- `name`
- `level`
- `exp`
- `nextExp`
- `state`
- `hp`
- `maxHp`
- `quick`
- `attack`
- `defense`
- `activeSkillIds`
- `passiveSkillIds`

本阶段新增的管理规则不修改模板数据，只修改玩家持有实例的状态。

## UI 规则

- 玩家界面只显示自然中文，不显示内部 ID、roll 值或自动测试字段。
- 宠物面板优先适配手机横屏；PC 使用同一模板。
- 面板使用较大的按钮，避免 hover-only 操作。
- 列表和详情都在一个面板内，避免来回跳转。
- 详情文字可以紧凑，但不能裁切关键状态。

## 操作规则

### 状态循环

一个状态按钮按固定顺序切换：

```text
休息 -> 战斗 -> 待机 -> 休息
```

因此：

- 休息宠点一次变战斗。
- 休息宠点两次变待机。
- 待机宠点一次变休息。
- 出战宠点一次变待机，不会自动让其他宠物接替出战。
- 当前可以没有出战宠物；下一场战斗只由人物参战，直到玩家把某只宠物切到战斗。
- 仓库宠暂时不能通过这个按钮切换状态。

### 休息宠

第一版只显示休息状态，不做“治疗/恢复”系统，也不在详情区额外补解释文字。

### 仓库宠

仓库宠显示在 `兽栏` 分组。

Phase39 起支持自由存入和取出。队伍满时不能取出兽栏宠，只提示 `队伍已满。`，不做交换或自动替换。

## 手工测试建议

启动：

```sh
godot --path client/godot --scene res://scenes/Main.tscn
```

建议检查：

- 点击地图界面的 `宠物`。
- 点 `休息布伊`，状态按钮只显示 `战斗`。
- 点一次状态按钮，休息布伊变为出战。
- 再点一次状态按钮，休息布伊变为待机，不会自动让其他宠物接替出战。
- 点 `黄色普通布伊`，详情显示 `10风`，不显示成长倾向。
- 对黄色普通布伊点两次状态按钮，列表中主宠切换为黄色普通布伊。
- 进入战斗后，我方前排主宠应变成刚刚设置的宠物。
