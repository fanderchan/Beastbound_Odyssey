# Phase 232：可扩展宠技目录与服务端被动目录基础

日期：2026-07-11

## 先复现的问题

当前七个宠技把目录字段 `slot=1..7` 全部占满，Godot、Node 工具和宠物模板检查又要求该字段全局唯一。因此第八个技能无论复用 1–7 还是填写 8 都会被拒绝。

更危险的是，实例虽然已经保存固定七格 `petSkillSlots`，服务端出招资格却读取全部 `activeSkillIds`。一旦目录超过七个，客户端可能提交“已经学会、但没有装备”的第八技能。

同时，Node 尚未严格加载 `battle_passive_skills.json`。现有五个被动只有 Godot 本地规则，不能作为联网战斗事实。

## 本阶段契约

### 技能身份与装备槽分离

- `battle_actions.json` 升为 schema v2；稳定 `id` 是目录身份。
- `preferredSlot` 只表示初始装配提示，可被多个目录技能复用；它不是技能编号，也不是实例装备状态。
- `activeSkillIds` 继续表示已学技能池，可以超过七个。
- `petSkillSlots` 是单宠实际装备栏，始终恰好七格、同一技能最多出现一次。
- 装配时先尊重实例已有槽位，再尝试首选槽，冲突时落入第一个空槽；没有空槽的已学技能保持未装备，不会被删除。
- 服务端战斗命令资格改为只读取规范化后的已装备技能，不能使用未装备技能。
- legacy `slot` 只在纯兼容 helper 中可读；正式 schema v2 目录会明确拒绝该字段，避免新旧语义再次混用。

### 严格服务端被动目录

新增 focused `battle-passive-catalog.js`，启动 `createAuthService()` 时只加载一次共享被动 JSON，并严格验证 ID、状态、元素、抗性范围和支持的效果字段。缺档或损坏会启动失败，不会静默退化为空目录。

纯规则已锁定当前五项事实：

- `bui_resistant_skin`：每点地/水/火/风分别提供 1% 石化/中毒/混乱/睡眠抗性；
- `wuli_hard_shell`：每点地提供 10% 石化抗性，达到 100% 时免疫；
- `stone_immunity`：石化免疫；
- `poison_resistance`：中毒抗性至少 35%，与既有来源取较大值而非叠加；
- `quick_instinct`：当前数据 `effect={}`，保持明确 no-op，不在没有产品公式时暗加速度或先手。

未知旧被动不授予效果，只返回内部诊断；不会把未知 ID 或目录内容写进玩家 UI。

## 本阶段明确不做

- 本提交不把被动效果接入正式联网 actor，因此不改变任何现有战斗胜负或状态命中率。
- 本提交不启用服务端闪避、幸运一击或反击。
- 不新增可获得宠技、不改变训练价格、不修改宠物数值、成长、捕捉、转生或旧存档。
- 不生成或替换美术；以后反击/闪避先复用现有 attack/dodge/hurt 表现，正式动作需求另行验收。

## 后续接线门槛

P0.4 下一小步必须让服务端从 `formId` 派生元素和种系被动，在 actor 进入房间/换入战斗时一次性生成抗性与免疫；再以私有不可预测种子结算闪避、幸运一击与单层反击。客户端只回放服务端最终事件，不能二次重掷或额外合成反击。

## 验证

- 目录工具用合成第八技能证明目录可达 8，而实例仍只装备 7；
- Godot 同一夹具证明 schema v2、重复 `preferredSlot`、七格装配和既有训练/战斗菜单兼容；
- Node loadout 纯测试覆盖八个已学技能、七个装备槽、显式换槽和 legacy 读取；
- Node 被动测试覆盖五项当前定义、未知被动及严格坏数据拒绝；
- 既有战斗房间 58 项、定向组合 66 项及完整服务端 304 项全部通过，证明本阶段没有破坏既有结算和服务端领域。

这一步只建立安全地基；P0.4 仍未完成，不能宣称联网被动、闪避、幸运一击或反击已经生效。
