# Phase 233：服务端宠物被动物化与私有战斗随机基础

日期：2026-07-11

## 先复现的问题

P0.4a 虽然让 Node 严格加载了五个共享被动，但联网 actor 仍只携带 `passiveSkillIds`，没有从服务端 `formId` 派生元素、抗性和免疫。因此布伊/乌力的被动在真实联网状态结算中仍不生效。

另一方面，现有 `room.seed` 会下发给客户端。它可以继续承担旧表现或兼容用途，却不能成为未来 PvP 闪避、幸运一击和反击的随机秘密。

数值审计还确认：`combat_formulas.json` 中写着的 3% 基础闪避、5% 基础暴击和 1.35 倍伤害目前并不属于默认 legacy 实战。P0.4 如果直接照这些说明接入，会从“权威化”变成一次未评审的平衡重做。

## 本阶段完成的合同

### 服务端一次性物化宠物被动

新增 focused `battle-actor-rules.js`：

- 只按服务端 `formId` 解析 `pet_templates.json`，不信任 actor/profile 提交的 `elements`、`lineId`、`statusResist` 或 `statusImmune`；
- 合并模板种系被动与服务端宠物实例已有的继承被动；
- 清空派生结果字段后统一调用严格 `battle-passive-catalog.js`；
- 未知旧被动不授予效果，只保留在内部诊断，不进入玩家事件或 UI；
- 玩家宠、野宠、训练伙伴宠、庄园战宠通过同一个房间 actor 入口物化；换宠时对新上场宠再次物化。

真实房间检查已经证明：

- 火 10 布伊获得 10% 混乱抗性；
- 地 10 乌力获得 100% 石化抗性和石化免疫；
- 换入的地 10 乌力仍使用同一事实；
- 对它施放石化凝视时，服务端事件稳定返回 `statusResult=immune`，目标不会被写入石化状态。

`quick_instinct` 仍保持明确 no-op。没有产品公式前不会暗加速度或先手。

### 私有战斗随机权威

新增 `battle-random-authority.js`：

- 每个房间在服务进程闭包的 `Map` 中拥有独立 32 字节 CSPRNG secret；
- 以 HMAC-SHA256 和固定字段上下文生成万分位 roll；
- `dodge.v1`、`critical.v1`、`counter.v1` 等 purpose 域相互隔离；
- 房间关闭立即释放 secret；服务重启本来就会清空 runtime-only 房间；
- secret 不写入 room、battle、event、trace、snapshot 或持久化数据；
- 缺少已注册房间时失败关闭，绝不回退公开 `room.seed`。

当前三个建房入口——切磋、队伍 PVE、庄园战——均已注册私有随机生命周期。

### 反应公式先锁 parity，不提前启用事件

新增纯 `battle-reaction-rules.js`，锁定当前默认 legacy 的真实合同：

- 闪避、幸运一击、反击使用现有平方根敏捷对抗及原有上下限；
- `luck` 和各 bonus 都按“百分点”解释；
- 幸运一击伤害保留“基础伤害 + 目标防御破防追加”的既有取整顺序；
- 反击伤害为反击者普通攻击伤害的 75%；
- 普攻可闪避/幸运一击/反击，伤害宠技可闪避/幸运一击但不反击，合击三者都不触发；
- 反击本身可被闪避或触发幸运一击，但绝不产生第二层反击；
- 保留当前已测试行为：普通攻击即使被闪避，存活且未被石化/睡眠的目标仍可能反击。

本阶段没有把三类反应事件接入联网回合。原因是协议 v3 客户端会把服务端 `damage=0` 夹成 1、忽略幸运一击并丢弃反击事件。服务端启用必须与协议 v4 和 Godot 纯回放作为同一部署组完成。

## 兼容、性能与遗留边界

- 不修改存档结构、宠物成长、技能价格、捕捉、转生或 HTTP 请求字段。
- 被动物化发生在建房/换宠，不进入逐帧或每次绘制热路径。
- 本阶段会真实改变布伊/乌力在联网状态技能中的结果，这是 P0.4 的目标行为，不是影子模式。
- 公共 `room.seed` 暂不删除，避免破坏旧客户端和现有目标/表现兼容；后续竞技随机只允许走私有 authority。
- 联网中毒回合伤害目前还缺服务端 `status_tick` 事件；这不是本阶段顺手扩大的内容，已在统一路线图 P0.4d 记录。
