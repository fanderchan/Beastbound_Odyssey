# Phase 283：联网自动捕捉设置权威保存与安全提示

## 玩家问题与本阶段目标

本阶段完成 `P1.1a`。此前自动捕捉面板可以修改选项，但联网账号没有可用的服务端写入口，也没有明确的保存动作；玩家会看到本地状态变化，却无法确认重新登录后是否仍然生效。复现时调用 `auto_capture_settings_update` 返回 `ok=false / profile_action_invalid`，档案 revision 保持 `0`，客户端也不存在保存按钮或成功回执。

本阶段让联网账号可以显式把捕捉设置保存到权威档案，并在面板中显示容量和成长判断提示。它不是自动丢弃实现：联网低战力丢弃继续隐藏且强制关闭，不会删除、吞掉或改写任何宠物。

## 权威设置合同

服务端新增 focused `auto-capture-settings` 规则，只接受 `auto_capture_settings_update` 的 `{settings: {...}}` payload，并统一规范化既有 12 个字段：启用状态、目标模式、目标形态、手工目标、血量、等级比较、等级值、捕捉工具、无目标动作、宠物技能槽、低战力丢弃和战力阈值。

- 形态 ID 必须存在于服务端宠物目录；捕捉工具复用服务端既有 allowlist；数值按客户端既有范围收敛。
- 未知 settings 字段被丢弃，未知顶层字段、旧 `params` 别名和完整档案上传均被拒绝。
- 玩家即使伪造 `autoDiscardLowPower=true`，服务端写入值仍固定为 `false`。
- 成功写入复用既有档案持久化和 revision 机制，响应返回更新后的权威档案；重建 service 后仍能重新拉取相同设置。
- 默认设置与玩家更新共用同一规范化器，避免客户端、默认档案和写入口逐渐产生三套规则。

没有修改 HTTP/WS 协议版本、档案字段名、数据库 schema、宠物成长数值或旧宠实例。旧档缺少/损坏设置时仍按既有默认值归一化。

## 客户端体验与安全边界

自动捕捉页新增“保存捕捉设置”按钮。联网修改只改变当前编辑状态，点击保存后才调用权威 action；请求期间全部捕捉设置控件冻结，setter 也会拒绝飞行中的迟到编辑，避免旧响应覆盖新修改后仍显示成功。成功、失败和再次修改都有明确中文状态，失败不会伪装成已保存。本地档案继续保留既有自动保存，并可使用同一按钮主动保存。

面板同时显示队伍加兽栏的 `5 + 20` 总容量、已用和剩余格数；剩余两格或已满时给出预警。成长提示明确要求先看公开 Lv1 四维、再训练到约 Lv20 判断去留，不使用或泄露隐藏成长底档。联网模式继续隐藏低战力丢弃开关，避免玩家误以为尚未具备收容、恢复和审计的删除功能已经安全可用。

所有新增计算只在打开/刷新面板或点击保存时执行，没有进入 `_process`、`_draw`、HUD signature 或移动输入热路径。
