# Phase 284：公开事实自动捕捉筛选合同

## 玩家问题与本阶段目标

本阶段完成 `P1.1b`。此前联网自动捕捉只有形态、血量、等级和工具等基础设置，不能按宠物系别、元素、是否新图鉴、同形态持有数量或公开 Lv1 四维筛选；更重要的是，筛选如果由客户端单独判断，改包客户端可以伪造目标事实，且很容易错误读取尚未公开的隐藏成长。

本阶段把自动筛选做成服务端权威的两段合同：投网前只看玩家已经看得到的战斗和图鉴事实，捕获成功后只看该冻结个体的公开 Lv1 四维。它不是自动丢弃功能；无论抓后命中、未命中或资料异常，宠物都完整保留。

## 设置合同与兼容规则

权威 `autoCaptureSettings.filterPolicy` 使用 schema 1，包含：

- `lineIds`：宠物系别白名单；空数组为不限。服务端支持多个，当前 PC 面板提供单系别选择。
- `element`：火、水、地、风的任一/同时匹配及最低属性点；不勾元素时不限。
- `onlyNewCodexForm`：仅捕捉尚未收入图鉴且本场尚未捕获的形态。
- `maxOwnedSameForm`：捕捉前的同形态上限，`0` 为不限；持有数量加本场待结算捕获共同计数。
- `levelOneFourV`：生命、攻击、防御、敏捷的最小/最大值，任一边界为 `0` 时该边不限。

新客户端提交的策略必须字段完整、schema 正确、ID 存在且数值合法；未知系别/元素、反向范围或越界值明确拒绝。损坏旧档会保守修复。旧客户端没有提交 `filterPolicy` 时保留账号已有策略，避免旧版本保存基础设置时把新筛选清空。

客户端提供形态、系别、元素组合、仅新图鉴、同形态上限和四项 Lv1 范围控件；嵌套修改不会覆盖其他捕捉设置，请求 pending 时全部控件冻结。提示文字明确说明 Lv1 四维只能抓回后评价，当前不会自动处理宠物。

## 服务端权威时序

自动流程才发送 `captureMode=auto`；手动点击捕捉不携带该标记，继续遵循原有捕捉规则。

1. 服务端先完成账号、回合、操作者、目标和基础捕捉资格校验。
2. 在读取捕捉候选、增加 attempt、抽成功率或消耗捕捉网之前，服务端用当前目标 HP、等级、形态、权威显示名、目录系别/元素、图鉴历史、持有数量和本场待结算数量重算公开筛选。
3. 指令入队和真正执行事件时各复核一次，避免等待期间目标或数量变化后仍按旧结论投网。
4. 只有筛选通过才进入既有捕捉候选与随机数流程。未命中或资料不可判断时不消耗工具、不增加 roll/attempt。
5. 捕获成功后沿用遇敌时已经冻结的同一个宠物个体，不重新抽品质。抓后筛选只接收 `initialStats` 与 `growthSpeciesLevel1Stats` 两份一致的公开 Lv1 四维；隐藏 seed、private roll、内置成长品质和最终预测值都不在可达输入中。
6. 抓后结果随公开捕获回执返回，客户端显示命中原因；所有状态固定 `retainPet=true`。

同场先捕获一只后，第二只同形态会被“仅新图鉴”拦截；该拒绝不会扣第二张网或增加第二只候选的 attempt。手工目标文本的公开身份集合也已统一：客户端和服务端都可按权威战斗显示名、形态、野外名、系别和亚型名称匹配，避免一端选中而另一端拒绝。

## 隐私、安全与非目标

- 纯规则测试用抛错 Proxy 锁死可读字段，证明投网前不读取候选宠物或随机结果，抓后不读取 hidden growth、seed、roll 或品质分。
- 服务端只把公开筛选事实和中文原因投影给客户端；未知 reason/status 不把原始内部字段显示给玩家。
- 本阶段不删除、丢弃、替换、降级或重抽宠物，也不开放联网低战力自动丢弃。
- 本阶段不实现满栏恢复、保护宠规则、最近处理记录和 GM 审计；这些属于 `P1.1c`。
- 本阶段不允许按隐藏成长趋势筛选；只有玩家取得足够公开观察证据后的规则才属于 `P1.1d`。

新增计算只发生在选择自动目标、提交/执行捕捉指令、捕获结算和打开设置面板时，没有进入 `_process`、`_draw`、HUD signature 或移动输入循环。

## 验证证据

- 公开筛选纯规则与内存服务集成 `10/10`：覆盖组合命中、公开字段缺失、同场第二只同形态、手动捕捉旁路、筛选失败零扣网/零 attempt、冻结个体原样获得、抓后四维不命中仍保留，以及 hidden seed/roll 零读取和零公开。
- 权威设置与本机回环 HTTP `7/7`；既有捕捉存入、冻结候选不重抽、满容量投网前拒绝回归 `3/3`。
- Godot parse 无脚本错误；自动捕捉设置与捕获反馈定向检查 `2/2`，最终日志明确 `status=ok`、`manual_identity=true`、`auto_payload=true`。同时修正旧 QA 对不存在 host facade 的调用，避免退出码 0 掩盖运行时错误。
- Pet Design Inspector：`errors=0`，保留两项既有 warning；本阶段没有改变宠物成长档、技能、目录或世界投放。
- headless idle `process_total` 约 `0.17–0.44ms`；真实移动输入压力约 `0.23–0.33ms @ 60FPS`，`movement status=ok`，未见新增筛选进入热路径。
- Node 语法、Pet Design Contract、`git diff --check` 均通过。

全部服务端验证使用内存 store 或本机临时回环 HTTP；未连接共享 MySQL，未读取或修改真实玩家数据，也未启用任何自动删除。

## 后续边界

`P1.1` 仍未完成。下一片 `P1.1c` 先建立可恢复临时收容、保护规则、最近处理记录和 GM 审计：任何自动移出都必须先保存完整私有宠物快照，绑定、任务、付费、锁定、骑乘、已培养和来源不明宠必须失败关闭。
