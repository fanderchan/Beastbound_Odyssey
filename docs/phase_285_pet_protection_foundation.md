# Phase 285：宠物来源与保护判定地基

## 玩家问题与本阶段目标

本阶段完成 `P1.1c-1`，不开放自动删除。

调查确认，联网捕捉的正常满栏路径已经会在抽取成功率和消耗捕捉网前停止；真正的丢失风险来自两条遗留路径：本地兼容结算在满栏时只返回公开的 `lostCapturedPets` 而不入档，旧低战力规则还会直接把新宠放入 `autoDiscardedPets`；另一方面，绑定宠和已进入容量竞态兜底的宠物仍可通过兽栏清理永久删除。服务端丢宠路径也缺少已有的任务保护复核。

本阶段先建立统一的失败关闭保护规则，给今后可恢复收容和自动处理提供唯一安全入口；同时封住已经存在的不可恢复旁路。

## 服务端可信来源

- 只有冻结捕捉候选在服务端 `materialize` 成功后，才写入私有 `source="wild_capture"`。
- 玩家不能在捕捉命令中提交来源；普通档案 action 也没有修改来源的入口。
- 历史宠物不按实例 ID、形态或外观猜来源。缺失、损坏、GM、教学、奖励、商业或其他来源全部视为不可信，不能进入自动处理候选。
- `source` 和 `captureOverflowPending` 不进入 `publicPet` 原始字段投影；后续玩家 UI 只显示服务端推导的保护原因，不泄露内部来源凭据。

## 统一自动处理保护器

新增纯服务端规则 `pet-protection-policy.js`。它既提供严格纯规则，也提供权威 profile fact builder：从档案骑乘/出战 ID、当前任务判断和服务端宠物模板计算技能新增、遗忘、继承或槽位变化，调用方不能自行把这些事实填成安全。兽栏清理和地面丢宠已经通过该入口复核相应保护原因。只有同时满足以下条件的宠物才返回 `eligible`：

- 来源严格等于服务端写入的 `wild_capture`；
- Lv1、零经验、无培养、无转生辅助或技能变化证据；
- 未锁定、未绑定、未珍藏；
- 非骑乘、非出战、非当前任务所需；
- 不在容量竞态待整理状态。

任意字段缺失、类型损坏、未来培养字段未识别或调用事实不完整都会失败关闭。结果使用稳定原因 code 和中文说明，并深冻结，不能被调用方改写。

规则不读取 `petGrowth`、private seed/roll、品质、隐藏成长底档或 Lv140 预测。测试用抛错 Proxy 覆盖这些字段，证明保护判定不依赖玩家尚未知道的成长质量。

## 本小片已封住的不可恢复旁路

- 本地兼容结算不再执行低战力自动丢弃；即使旧设置仍为开启，新抓宠也完整保留，`autoDiscardedPets` 为空。
- 本地满栏捕获不再只返回 `lostCapturedPets`；冻结个体以超额兽栏宠保存并标记 `captureOverflowPending`，日志明确“完整保留为待整理宠物”。这是正式收容完成前的保命兜底。
- 服务端和客户端兽栏清理都拒绝绑定宠与 `captureOverflowPending` 宠；服务端同时补齐任务宠和骑乘宠检查。
- 服务端地面丢宠补齐当前任务保护；绑定兼容检查同时识别 `binding`、`bound` 和 `bindingLocked`。
- 新增公开投影回归，确认私有来源和待整理标记不会作为原始内部字段泄露。

## 非目标与后续边界

- 本阶段没有启用联网自动删除、低战力丢弃、按成长趋势处理或任何付费宠推断。
- 当前仍有一条已确认的服务端风险：候选已经 claim 后若私有 `materialize` 异常，旧结算会跳过该宠物而继续关闭战斗。它需要 `P1.1c-2` 的正式私有快照收容或结算失败恢复，不能用公开摘要伪造宠物；在完成前不得开放任何联网自动处理。
- `captureOverflowPending` 仍是最后兜底，不是正式收容 UI。`P1.1c-2` 将把已认领后的容量漂移写入档案内完整私有快照收容，并提供有空位才可执行的幂等取回；收容失败仍保留当前超额宠兜底。
- `P1.1c-3` 才增加最近处理、玩家保护说明和授权 GM 查询/恢复入口。
- 历史版本真正丢失且没有完整私有快照的宠物不能伪造恢复；只能留作审计或人工补偿问题。
- 手工清理普通未绑定宠物的产品规则没有在本阶段扩大；已培养、商业、来源不明等限制首先约束未来自动处理。正式收容和删除确认会在后续小片统一收口。

## 验证证据

- Pet Design Contract：`p1_1c_pet_safety_foundation` 校验通过；Pet Design Inspector `errors=0`，保留两项既有 warning。
- 纯保护规则与 profile fact builder `7/7`：覆盖可信新鲜野宠、身份/字段损坏、未知来源、锁定/绑定/珍藏、骑乘/出战/任务、竞态待整理、等级/经验/培养/转生辅助/技能变化、多原因顺序、真实模板技能差异，以及隐藏成长零读取。
- 服务端定向 `4/4`：冻结宠 materialize 原样增加可信来源；绑定蛋宠无法清理；待整理溢出宠无法清理；公开投影不泄露内部来源/标记。
- Godot 定向检查：自动捕捉与宠物管理安全检查均 `status=ok`。满 `5+20` 后本地冻结宠完整写入第 26 个待整理实例，`lostCapturedPets=0`；旧低战力开关下仍保留宠物且不出现“已自动丢弃”。绑定宠和待整理宠客户端清理均被拒绝。
- Godot parse、Node 语法、战斗动作目录与 `git diff --check` 通过。
- 性能抽样没有新增每帧规则：idle `process_total` 约 `0.23–0.52ms`；真实跨帧移动约 `0.21–0.39ms @ 60 FPS`，`movement status=ok`。

全部服务端测试使用内存 store；未连接共享 MySQL、未读取或修改真实玩家数据，也未修改数据库全局或会话参数。
