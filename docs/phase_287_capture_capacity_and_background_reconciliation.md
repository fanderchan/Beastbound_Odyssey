# Phase 287：捕捉容量前置提示与后台自动回收

## 玩家问题与产品判断

本阶段收口 `P1.1c-3` 的正常玩家边界。玩家真正需要解决的是：随身宠和兽栏都满时，不应还能选择捕捉、浪费回合或捕捉工具；服务异常后遗留的完整私有宠物快照，也不应要求玩家理解一个技术性的“安全收容”系统并亲自领取。

石器时代 8.0 本地参考的已验证行为是：捕捉成功路径创建宠物时只寻找 5 个随身宠位置，没有空位就失败；它没有把异常恢复做成常驻收容面板，宠物邮件同样不能绕过满位。Beastbound 与其存在两项明确差异：

- Beastbound 还有 20 格兽栏，并允许长时间自动捕捉；随身满后自动进入兽栏是有价值的现代化减负。
- 服务端已经在捕捉成功的同一持久化提交内保存冻结宠物快照；进程故障后的极少数 pending 是技术恢复事实，不是新的日常玩法。

因此本阶段保留石器的“满位不能捕捉”玩家心智，并把判断提前到战斗 UI；后台快照只做自动、幂等和不可复制的恢复。常驻“安全收容/待取回”玩家入口被判定为过度产品化，不再开发。

## 玩家可见容量合同

- 联网战斗的捕捉区显示 `随身 x/5、兽栏 y/20`；本场本人已经捕获但尚未结束结算的宠物也计入占用，并按服务端相同顺序先填随身、再填兽栏。
- `5/5 + 20/20` 时显示“已满”，人物主菜单的捕捉入口和捕捉子菜单里的空手/全部捕捉工具均禁用；返回、攻击及其他菜单不受影响。
- 捕捉入口、工具选择和目标选择各自复核同一容量模型，防止脚本调用或 UI 状态刚变化时绕过灰按钮；服务端仍在 roll 和扣工具前做最终权威复核。
- 客户端只使用服务端公开档案与当前服务端房间，不轮询、不发额外容量请求、不读取私有收容字段。账号、房间或档案尚未同步完整时失败关闭，并只显示玩家可理解的同步提示。
- 这项改动不改变捕捉率、1 级 4V、隐藏成长、逃跑概率、宠物落位顺序或自动筛选规则。

## 后台自动回收合同

- 正常战斗结束仍在既有结算中把 pending 原样放入随身或兽栏；只有服务重启等导致运行期房间消失后，才需要后台回收。
- 官方客户端登录及事件流重置后都会拉取本人档案。服务端只在“本人没有活动战斗、私有 pending 合法、宠物总容量确有空位”三个条件同时满足时执行稀有 read-repair。
- 多条 pending 按稳定捕捉序号、创建时间和 recovery ID 排序，能放入的宠物在一份候选档案中统一恢复；无论恢复一只还是多只，只增加一次 profile revision、只保存一次。
- 容量不足时按顺序恢复到满为止，其余完整快照继续保留；满位账号的普通档案拉取保持纯读，不制造无意义写入。玩家之后腾出位置并再次安全拉档时继续自动恢复。
- 恢复过程复用 Phase 286 的完整快照、身份冲突检查和完成墓碑。同一 GET 重试、提交结果模糊或多节点竞争都不能生成第二只宠物；任一结构/身份异常时整份候选修改不发布，留给授权 GM 审计。
- 活动战斗中的 pending 绝不被档案拉取提前搬走，避免和当前房间结算重复。

## 权威、隐私与性能边界

- `petRecoveryShelter` 继续完全排除在公开 profile、战斗广播、客户端缓存和玩家 UI 之外；容量行只展示本人可从公开宠物列表和本场捕获推导的数字。
- 正常 `GET /profiles/me` 继续走既有 pure-read fast path。只有确实存在可搬移的孤立 pending 时才进入持久化候选与 COMMIT-before-response 路径，因此不会给日常登录或战斗增加稳定写压力。
- 不新增数据库表、协议版本、网络轮询、每帧扫描或 MySQL 全局/会话参数；不连接或改动真实玩家数据库。
- 既有按 recovery ID 的权威 API 继续作为服务端/测试兼容地基，但不提供普通玩家常驻入口；按账号、收容 ID、宠物 ID 的授权 GM 审计与人工恢复仍是后续运营工具，不是本阶段玩家功能。

## 验证

- 服务端纯规则：多条稳定顺序、随身优先/兽栏其次、部分容量、一次性完成墓碑、损坏/冲突原子失败。
- 服务端档案与 HTTP：活动战斗不回收；重启后下一次档案拉取自动回收；多只只增一次 revision；满位零写；响应与持久回执零私有字段；持久化失败不得提前返回成功。
- Godot：5+20、5+19 加本人本场 pending、他人 pending、4+20 加 pending、4+19 加 pending、房间/账号缺失失败关闭；主捕捉与全部工具禁用但返回、攻击、宠物/精灵/物品复用槽位正常。
- 1280×720 截图确认容量行、满位状态和禁用可读性；Godot parse、相关捕捉检查、`git diff --check` 和必要的 idle/moving 性能探针通过。

## 验证结果与剩余边界

- Node 语法通过；收容纯规则、服务档案和回环 HTTP 定向测试 `17/17` 通过，覆盖稳定顺序、部分容量、一次 revision、活动战斗不回收、满位纯读、冲突原子失败、COMMIT 前不返回以及写失败不发布。
- Godot parse、`--auto-capture-tools-check`、`--auto-battle-capture-check` 共 `3/3` 通过。容量矩阵、玩家主菜单、全部捕捉工具、物品复用槽位和资料未同步时的自动捕捉等待均为 `true`；额外的 parse 与 `--auto-pet-management-safety-check` `2/2` 通过，旧溢出保护提示不再向玩家展示“安全收容”技术名词。
- Pet Design Inspector 为 `errors=0`；两条既有 warning 是蓝人龙尚无显式世界遇敌位置和服务端完整档案尚未全局统一公开投影，不由本阶段引入。战斗动作目录检查通过。
- 正常 macOS 渲染路径自动保存并人工检查 `.run/phase_287_capture_capacity_1280x720.png`：分辨率 1280×720，容量行显示 `随身 5/5、兽栏 20/20（已满）`，捕捉禁用而攻击等指令可用，画面没有技术收容入口或私有字段。
- 本轮 fixed-fps headless 性能抽样：idle `process_total=0.03ms`；真实跨帧移动 `0.03–0.04ms @ 60FPS`，`movement status=ok`。容量计算只在战斗指令 UI 同步/提交时扫描最多 25 只公开宠物，不进入每帧热路径。
- 本阶段没有连接共享 MySQL、没有读取或修改真实玩家数据、没有修改协议或数据库参数。损坏记录、满位账号中已存在同宠物但仍残留 pending 等异常保持原样，等待 `P1.1c-4` 授权 GM 审计；不自动猜测或删除。
