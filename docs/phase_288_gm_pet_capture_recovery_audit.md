# Phase 288：授权 GM 捕捉恢复审计

> 当前策略更新（2026-07-17）：Phase 297 新增仅服务端使用的 `gm_pet_paid_reset_config`，现行 `local_qa_full_v2` 为服务端 11 项、客户端 29 项、双重授权 7 项；本页以下 `10/29/7` 保留为 Phase 288 当时的验收事实。

## 问题与产品判断

`P1.1c-1` 至 `P1.1c-3` 已经让正常捕捉在容量允许时直接落入随身宠或兽栏，并把服务异常期间的完整冻结宠物快照留在服务端私有档案中自动恢复。剩余问题只属于极少数异常运营场景：快照损坏、身份冲突、账号长时间满位，或自动恢复为了安全而失败关闭时，GM 需要知道是哪一条记录、为什么没有恢复，并能在条件恢复后人工重试。

石器时代 8.0 的成熟玩家心智仍是“宠物位满时不能捕捉”，没有必要为正常玩家增加一个技术性收容系统。Beastbound 之所以需要本阶段工具，是因为它采用服务端权威、COMMIT 前私有快照和自动捕捉，不是为了创造新的玩家玩法。因此本阶段只增加隐藏在 `GM/QA` 面板中的运营工具；普通玩家继续只看容量、捕捉结果和可理解的失败提示。

## 授权与请求合同

- 新命令固定为 `gm_pet_capture_recovery`，同时要求有效 GM 用户授权、该命令的服务端 grant、客户端本地 QA 插件和当前服务器授权摘要。隐藏按钮不能代替服务端鉴权。
- 共享本地策略升级为 `local_qa_full_v2`：10 项服务端命令、29 项客户端入口、7 项服务端权威客户端命令。旧 `v1` grant 或插件不会被静默接受。
- HTTP 入口为 `POST /gm/commands/gm_pet_capture_recovery`。请求正文只能包含 `action`、`targetUsername`、`recoveryId`、`petInstanceId` 四个字段；未知字段、双 selector、非法 ID 和无 selector 的人工恢复均拒绝。
- `search` 虽然不修改目标档案，但必须写 GM 审计，所以仍使用持久化幂等键，并且只有审计 COMMIT 后才返回成功。相同 operation ID 重放原响应，不重复写审计。
- `recover` 只接受一个精确 recovery ID，或一个只能命中唯一记录的 pet instance ID。宠物 ID 命中多条记录时要求改用 recovery ID，不能猜测。

## 查询与人工恢复

- GM 可只填目标用户名查询最多 50 条记录，也可用 recovery ID 或 pet instance ID 精确过滤。返回待恢复/已完成数量、目标当前 revision、是否在战斗、随身与兽栏容量，以及安全记录摘要。
- 安全摘要只包含状态、恢复 ID、宠物 ID、形态 ID、名称、等级、公开状态、捕捉序号、创建/完成时间和最终落位。目标内部 account ID、完整宠物对象、`petGrowth`、private seed/roll、技术容器和商业凭据不进入响应。
- 人工恢复复用既有 `recoverPetCapture` 原语：完整冻结个体原样进入随身栏或兽栏，图鉴同步登记；成功只增加一次目标 profile revision。同一 completed 墓碑再次执行只返回已完成，不复制宠物、不增加 revision。
- 对“宠物本体已经在档案里、pending 仍残留”的旧故障状态，恢复只把 pending 收尾为 `overflow_fallback` 墓碑并清除 `captureOverflowPending`，不会生成第二只；这一合法结果明确表示为 `changed=true + replayed=true`。之后再重放仍为零变更。
- 目标玩家仍在活动战斗、`5+20` 容量已满、记录不存在、pet ID 有歧义、身份冲突或收容资料损坏时失败关闭。候选档案必须先完成恢复后结构校验，才允许发布到目标档案。
- 每次成功、失败和查询都记录 GM 用户、命令、目标账号、selector、结果和必要的前后 revision。内部审计可保存 target account ID 用于追责，但该字段不返回客户端。

## 客户端与玩家边界

- `GM/QA` 面板新增目标用户名、恢复 ID/宠物 ID、查询和恢复控件。默认目标为当前 GM；恢复按钮在 selector 为空、无服务器授权或请求进行中时禁用。
- 查询/恢复响应不携带目标完整档案。只有 GM 恢复的是自己且目标 revision 确实变化时，客户端才安排一次本人档案重拉；恢复其他账号不会把对方档案装入 GM 客户端。
- 客户端对返回对象使用严格正向字段合同；多余字段、错误类型、非法 ID 或私有字段都会把整份摘要判为不可用。账号显示名、宠物昵称、ID 和服务端消息进入富文本前逐字符转义。
- 普通宠物面板、战斗捕捉 UI、邮箱和档案接口均未增加“安全收容”入口，也不会显示 recovery ID、审计 ID 或私有成长字段。

## 持久化、性能与非目标

- 命令进入现有 durable mutation 边界；目标档案变更、GM 审计和 operation receipt 在同一持久化候选中提交，HTTP 不在 COMMIT 前报告成功。失败候选不发布到运行时权威状态。
- 本阶段不新增表、协议版本、MySQL 参数、轮询或每帧逻辑。查询最多返回 50 条，并且只有 GM 主动点击时扫描一个目标账号的私有恢复集合；正常玩家和正常登录没有新增稳定开销。
- 本阶段不修复损坏数据、不删除墓碑、不绕过容量、不强制退出目标战斗，也不开放联网自动丢弃。成长观察成熟后的自动处理仍由 `P1.1d` 单独决定。
