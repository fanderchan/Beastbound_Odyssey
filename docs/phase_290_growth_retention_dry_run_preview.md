# Phase 290：Lv20 成长保留门槛与 dry-run 预览

## 结论

P1.1d-2 已把“训练到约 Lv20 后如何批量判断去留”做成可保存、可解释、但绝不执行处置的预览。玩家可以为综合成长以及生命、攻击、防御、敏捷分别设置最低成长分位；修改时客户端立即预演，联网保存成功后再显示服务器确认的同一份结果。所有路径固定 `retainPet=true`、`mutationCount=0`，不会移动、丢弃、覆盖或重掷宠物。

这个阶段没有阉割现有宠物面板的实测成长与 Lv140 外推。成长面板仍负责让玩家观察一只宠物；新预览只负责把玩家自己设置的保留门槛批量套到最多 25 只已有宠物上。

## 石器 8.0 基线、差异与是否过度设计

本地稳定石器 8.0 参考保留“玩家显式丢宠、满栏失败、服务端写丢宠日志”的简单心智，没有发现可直接照搬的隐藏成长自动删除合同。因此，Beastbound 不把自动筛宠伪装成石器原有机制，也不提前增加“安全收容”等普通玩家技术入口。

Beastbound 与参考的必要差异是：玩家预计长时间挂机、拥有 20 格兽栏，并把“抓 Lv1、练到约 Lv20、观察隐藏成长后决定去留”当成核心乐趣。批量预览能减少重复翻页和心算，值得保留；直接自动删除则会放大一次门槛误设的损失，当前不值得开放。为避免过度设计，本片只有五个整数门槛、一个结果区和既有保存按钮，没有规则树、权重评分、名单脚本或新增宠物容器。

## 规则合同

设置继续保存在既有 `autoCaptureSettings` 下，不需要独立表或破坏性迁移：

```json
{
  "growthRulePolicy": {
    "schemaVersion": 1,
    "overallMinimumPercentile": 91,
    "statMinimumPercentiles": {
      "maxHp": 90,
      "attack": 90,
      "defense": 0,
      "quick": 40
    }
  }
}
```

- 每个门槛只能是 `0..100` 的整数；`0` 表示该项不限。
- 全部已启用门槛都达到才预览为“将保留”，任意一项不足则预览为“进入待处理”。这比加权总分更容易理解，也不会让一项严重短板被另一项偷偷抵消。
- Lv1–19 只显示“观察中”；必须到 Lv20 / 19 次公开升级证据后才判断。
- legacy、资料损坏、形态/成长档错配或公开证据不足的宠物始终保留。
- 未配置任何门槛时功能关闭，不给宠物贴去留结论。
- 预览最多读取当前 25 个队伍/兽栏宠物，只使用公开 Lv1 四维、当前四维和共享万人分位表；不读取 private seed、private roll、内置品质或未来升级结果。

## 服务端与客户端边界

服务端对玩家新提交的 `growthRulePolicy` 做精确 schema 和整数范围校验；旧存档自动补成全 0，旧客户端漏交该字段时保留账号原值。既有 `auto_capture_settings_update` 在保存设置的成功响应里附带服务器计算的预览，响应仍经过公开结果投影；预览计算异常不会把已经验证的设置写入伪装成失败，也不会触发第二次资产变更。

客户端使用同一公开公式提供即时反馈，并逐项显示例如“综合 90%＜91%、攻击 91.9%≥90%”。玩家点击保存且收到成功响应后，标题才从“即时预览”变成“服务器已确认预览”。这既避免每次调一个数字都发网络请求，也不把客户端计算冒充最终权威结果。

客户端预览只在打开/刷新自动捕捉页或玩家修改五个门槛时运行；服务端预览只在保存设置时运行。两边都不进入 `_process`、`_draw`、HUD signature、移动输入或轮询热路径。

## 玩家界面

“内挂设置 → 捕捉”新增“Lv20 成长保留预览（仅模拟）”：

- 综合、生命、攻击、防御、敏捷五个 `0..100` 分位门槛；
- 明确说明 `0` 为不限、Lv20 前只观察、全部已启用门槛都达标才保留；
- 最多展示五只宠物的逐项原因，并给出全体数量汇总；
- 文案始终说明当前不会移动、丢弃或改写宠物；
- 联网低战力丢弃开关仍隐藏，服务端仍强制为 false。

## 兼容与非目标

- 不改协议版本；成功 action 只增加可选结果字段，旧客户端可忽略。
- 不改宠物实例 ID、成长随机数、四维、技能、状态、队伍位置或旧存档。
- 不新增或执行 MySQL 写入、不改共享数据库参数、不启动后端、不操作真实玩家账号或宠物；收尾只用既有 ops 状态命令只读确认后端仍停服。
- 不建立最近处理记录、恢复期或处置动作；这些属于 P1.1d-3。
- 不显示联网自动处理开关；P1.1d-4 的组合竞态和恢复门禁通过前继续关闭。

## 建议人工验收

这是玩家可见且涉及“是否留下宠物”的判断，建议用 `auth1373` 做一次短验收：

1. 启动本地后端并登录 `auth1373`；在 GM/QA 中选蓝人龙，使用已有 Lv20 个体，或“获取Lv1”后点“升1级”到 Lv20。
2. 打开该宠物的成长页，记下综合和四项成长分位；现有 Lv140 实测外推应继续存在。
3. 打开“内挂设置 → 捕捉”，滚动到“Lv20 成长保留预览（仅模拟）”。把某个门槛设为高于该宠实际分位 1 点，应立即看到“若开启，将进入待处理”以及对应 `实际＜门槛`。
4. 再把所有已启用门槛调到不高于该宠实际分位，应立即变成“若开启，将保留”。`0` 项应不出现在比较原因里。
5. 点击“保存捕捉设置”，确认标题变为“服务器已确认预览”。
6. 返回宠物列表并重新登录，确认宠物仍在、等级/四维/状态未变，门槛仍保留。
7. 如有 Lv19 个体，确认它只显示“成长观察中，Lv20 后判断”，不会出现待处理结论。
