# Phase 292：新鲜 Lv1 物种内 4V 分位筛选

## 结论

P1.1d-4 已把捕捉设置里的血、攻、防、敏绝对数值上下限替换为四个物种内 Lv1 最低分位。服务端只对“投网前冻结候选为 Lv1，且捕后仍是同形态、同等级、同一 authority-v1 Lv1 公开快照”的宠物计算；Lv2+、资料不足、档案错配或四维不一致全部保留人工判断。

这个判断只回答“这只宠的公开 Lv1 生命/攻击/防御/敏捷，在同种宠物中分别处于什么位置”。它不读取隐藏逐级成长，不预测 Lv140，也不改变“Lv1 4V 与隐藏成长完全独立”的产品规则。

当前命中和未命中都固定 `retainPet=true`。本切片给出权威评价和玩家反馈，但不偷偷开放自动放生；真正的自动处置必须先有玩家可见、可恢复的最近操作记录与 GM 审计合同。

## 石器 8.0 参考与必要差异

本地石器 8.0 参考支持简单的目标捕捉、满栏失败与显式丢宠，但没有发现一套可直接照搬的“按隐藏成长自动筛宠”合同。Beastbound 延续它容易理解的边界：捕捉专栏负责找目标、投网和刚捕获个体，不替玩家练宠。

Beastbound 的必要差异来自自己的核心玩法：不同宠物的 Lv1 数值尺度不同，而且 Lv1 4V 与隐藏成长独立。因此“攻击至少 14”不能跨物种表达同一种品质；改成“攻击至少本物种 85% 分位”后，所有物种共享同一套操作语义，同时仍保留各自成长档和追极品空间。没有增加规则树、权重总分或隐藏品质展示，避免过度设计。

## 分位合同

每项使用同一物种 authority-v1 成长档的公开 Lv1 生成分布：

```text
percentile(stat) = P(同物种随机 Lv1 的可见 stat <= 当前可见 stat)
```

- 分位是包含同值个体的离散 CDF，范围 `0..100`，数值越高越好。
- 生命、攻击、防御、敏捷分别判断；玩家设置的是四个最低分位，`0` 表示该项不限。
- 全部已启用项都达到才显示“已命中”；任意一项不足显示具体的实际分位和门槛。
- 计算只读取物种公开 `outputBase`、`initialOutputSpread`、`distribution`、`rareExtremeRate` 以及捕获个体公开 Lv1 四维。
- 不读取 `growthOutputSpread`、`innateGrowthBonus`、private seed/roll、综合隐藏品质或未来等级结果。
- `uniform`、`weighted_center`、`rare_spike` 三种现有分布均按成长权威算法的取整方式计算，不在线抽样。

服务端线上只在成功捕获 Lv1 后做一次小规模常数计算；没有数据库查询、JSON I/O、人口扫描或每帧逻辑。全物种万人抽样只是离线审计工具。

## 身份与失败关闭

服务端在捕后评价前依次确认：

1. 投网前评价确实命中，且记录了服务器解析的形态和等级；
2. 捕后宠物形态、等级与投网前完全一致；
3. 捕获等级恰为 Lv1；
4. 当前血攻防敏、`initialStats` 与 `growthSpeciesLevel1Stats` 三份公开事实完全一致；
5. 成长模型是 `pet_growth_authority_v1`，宠物档案 ID 与服务器按形态解析的档案一致。

任一事实缺失或异常只返回“无法判断并保留”，不会退回客户端计算、使用历史高等级宠的 Lv1 记录冒充新鲜捕获，也不会读取私有候选。Lv2+ 在读取任何 Lv1 四维或成长字段前直接返回 `manual_review`。

## 设置 schema 与旧存档

新设置为 schema v2：

```json
{
  "filterPolicy": {
    "schemaVersion": 2,
    "lineIds": ["man_dragon"],
    "element": {"mode": "all", "ids": ["water"], "minPoints": 10},
    "onlyNewCodexForm": false,
    "maxOwnedSameForm": 10,
    "levelOneMinimumPercentiles": {
      "maxHp": 90,
      "attack": 85,
      "defense": 0,
      "quick": 40
    }
  }
}
```

- 四项必须完整，且只能是 `0..100` 整数；未知字段、缺项、小数和越界值明确拒绝。
- 旧客户端提交的 schema v1 仍被接受，系别、元素、新图鉴和同形态数量规则继续保存。
- 旧的绝对数值上下限无法可靠换算为跨物种分位，因此四项安全迁移为 `0`，不会误把旧“攻击 14”解释成“攻击 14%”后开始筛宠。
- 不改协议版本、宠物实例、成长随机数、账号 revision 语义或 MySQL schema；没有连接共享数据库或操作真实玩家数据。

## 建议人工验收

这是玩家可见且关系到“抓到后是否值得看”的高价值反馈，建议用本地 GM 号短验一次：

1. 启动本地后端并登录现有 GM 号 `auth1373`，确认宠物栏或兽栏至少有一个空位。
2. 打开“内挂设置 → 捕捉”，把目标设为蓝人龙、等级设为 `= Lv1`，分位门槛可先设生命 90、攻击 85、防御 0、敏捷 40，保存。
3. 通过 GM/QA 或真实遇敌捕获一只 Lv1 蓝人龙；确认成功反馈同时显示原始初始四维和四项“物种内分位”，且没有 Lv140/隐藏成长结论。
4. 把其中一项门槛调到高于刚才显示的分位，再抓一只 Lv1；应显示“未命中”和具体低于门槛原因，但宠物仍完整进入队伍或兽栏。
5. 把捕捉等级改为 `= Lv20` 并捕获高等级蓝人龙；即使四项门槛全设 100，也应显示“需人工判断、Lv2+ 默认保留”，不能套用 Lv1 分位自动结论。
6. 重新登录，确认 schema v2 门槛仍保存，所有已捕宠物的等级、初始四维、当前属性和位置都未改变。

若只想看界面，不必启动后端，可直接查看上述 1280×720 截图；功能性手工验收才需要 `auth1373`。
