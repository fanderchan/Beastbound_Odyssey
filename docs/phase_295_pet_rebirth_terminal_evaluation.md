# Phase 295：普通宠转生阶段评价与二转终局档位

## 结论

P1.2b 修复的是评价口径，不改 P1.2a 已落地的转生随机池、消耗、成长结果或旧宠数据。

P1.2a 加入 Lv80→Lv140 准备倍率后，成长页仍使用早期手写绝对阈值：Lv140 四满石的一般运气中位结果，一转 `1.540/级` 被误报为 `96.25% / S`，二转 `1.760/级` 被误报为 `95% / S`。这会让绝大多数认真准备的普通结果看起来都是极品，也无法判断两次转生合计处于什么档位。

当前改为两条互不替代的证据：

- **本次运气**：沿用服务端确认转生时已经公开记录的随机百分位与评级，只回答“这次抽签运气如何”。
- **实际成品增量**：把真实得到的一转增量、二转增量及两转合计，与 `Lv140四满石全物种基准` 比较，只回答“最后写进宠物的成长值处于什么档位”。

因此少喂石但抽到 S 时，玩家会同时看到“运气 S”和较低的成品增量档；满准备的中位结果会回到约 `50% / C`，不再假装是 S。

## 石器 8.0 基线与 Beastbound 差异

本地稳定 StoneAge 8.0 参考的 `npc_transmigration.c` 只负责选择、确认、条件检查和结果生成；`char_data.c` 根据 MM、目标宠、目标等级和转数计算结果，并用服务端上限约束一转/二转。原版没有额外的 S/A/B/C/D 自动评价面板。

Beastbound 不为此新增 NPC、独立面板、进度条、货币或操作步骤。保留现有宠物成长页的 `0转成长 / 1转成长 / 2转成长` 三个页签，只修正已有评级，并在二转页把两转合计提升到表格第一行。这个差异有明确项目理由：本项目把转生随机、隐藏成长观察和极品培育作为核心长期玩法，玩家需要判断结果，但不应被错误阈值诱导。

## 单一评价来源

共享 `client/godot/data/balance/pet_rebirth_balance.json` 升级为 `pet_rebirth_balance_v3`，新增 `pet_rebirth_evaluation_v1`：

- 玩家可读的基准名称；
- 基准目标等级、每项石头点数、成长档选择规则和样本数；
- 一转/二转的总内部成长与血攻防敏单项分位锚点；
- 普通二转两次合计的总内部成长与四项分位锚点。

Node loader 对评价版本、参考条件、样本门槛、字段完整性和 `min→p25→p55→p85→p95→max` 单调性严格校验并冻结。Godot 目录校验还要求配置中的 `profileCount` 与当前全部非 MM 成长档实时一致；以后新增宠物成长档时如果没有重跑审计并更新阈值，客户端门禁会直接失败，不允许旧基准静默漂移。

## 校准模型

`tools/pet_rebirth_evaluation_audit.mjs` 使用当前全部 30 个非 MM 成长档，每档 10,000 组，固定基准为：

- 目标宠 Lv140；
- 一转与二转 MM 四项都喂满 50 点；
- 两次随机百分位相互独立；
- 二转四维分配读取已经包含一转加成的目标成长；
- 血量按 `4:1` 折算内部成长，与现行宠物战力权重一致。

共得到 300,000 组完整两转样本。总内部成长锚点为：

| 对象 | 最低 | P25 | P55 | P85 | P95 | 最高 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| 一转增量 | 1.265 | 1.403 | 1.567 | 1.732 | 1.787 | 1.815 |
| 二转增量 | 1.485 | 1.623 | 1.787 | 1.952 | 2.007 | 2.035 |
| 两转合计 | 2.750 | 3.137 | 3.326 | 3.551 | 3.674 | 3.846 |

评级继续使用既有档位：`D <25%`、`C 25..55%`、`B 55..85%`、`A 85..95%`、`S ≥95%`。单项血攻防敏使用同一批样本各自校准，不拿攻击阈值套生命或敏捷。

审计得到的阶段分布严格为 `D 25% / C 30% / B 30% / A 10% / S 5%`；两次独立结果求和后因卷积和离散取样形成 `D 24.50% / C 29.81% / B 30.68% / A 9.77% / S 5.24%`，每档偏差均低于 1 个百分点。四维折回总内部成长的最大守恒误差为 `1.34e-15`。

## 旧档与隐私

- 既有宠物的四维、成长加成、转数、随机结果和历史记录全部不改，不迁移、不补差、不重抽。
- 新评价只读取玩家原本已经可见的 `visibleGrowthBonus`、累计 `rebirthGrowthBonus` 及公开随机档。
- 旧二转宠即使历史记录无法拆出第二次单独增量，只要累计转生成长仍在，仍可得到两转总评价；阶段页明确提示无法拆分，不伪造历史。
- 私有随机 seed、基础内部池、准备倍率、MM 隐藏成长和服务端审计字段仍不进入普通玩家投影。

## 验证

- Node：共享配置严格加载、评价纯函数、权威成长周期、档案动作和隐私投影定向 `47/47`。
- 数值：既有三等级 900,000 组转生池审计继续通过；新增 30 档 × 10,000 组终局评价审计通过。
- Godot：balance catalog、balance digest/version、MM 流程/公式、培养界面和成长观察定向检查通过；成长观察覆盖阶段运气、实际增量、二转总评、旧二转累计回退和首屏表格文字。
- 真实客户端：1280×720 Metal 截图通过，`二转总评` 与 `运气` 均在首屏可读。
- 性能：idle 稳定段 `process_total=0.33..0.45ms`；317 次跨帧鼠标移动输入 `status=ok`、平均 `13µs`、最大 `154µs`，稳定 60 FPS 时 `process_total=0.34..0.38ms`。评价只在宠物成长页刷新时计算，不进入 `_process`、`_draw` 或输入热路径。
- 未运行完整本地 CI；未连接 MySQL、未启动玩家后端、未操作真实玩家数据。
