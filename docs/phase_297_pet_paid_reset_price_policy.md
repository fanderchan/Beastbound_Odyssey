# Phase 297：全形态付费重置价格与 GM 策略目录

## 结论

P1.2c-2 已建立“报价事实”，但没有提前实现玩家扣款或宠物重置：

- 当前 `pet_templates.json` 的 32 个形态全部且只能拥有一条服务端重置价格规则；缺少、重复或未知形态都会让严格目录失败；
- 普通宠、捕捉宠、奖励宠、未来商业/进化/融合宠都使用同一套报价合同，不以“是否商业宠”决定能不能重置；
- 价格只由形态默认档或 GM 覆盖决定，不读取该宠的 Lv1 4V、隐藏成长、观察评级、重置次数或充值记录；
- 石币与非商业钻石档默认允许绑定、非绑定合并，先扣绑定；商业档默认只允许非绑定钻石；
- GM 可整档覆盖或单形态覆盖 `currencyId + amount + walletPolicyId`，单形态优先，并使用 `expectedRevision` 防止并发覆盖；
- 配置写入现有 MySQL `server_state/auth` 文档，只有所属 durable save 成功后才发布；旧存档没有该字段时按 revision 0 默认目录读取；
- 成功重置固定为完整回到 Lv1/0转、无限次、解绑且不返还投入。这些产品不变量不能被 GM 价格配置改写。

本阶段没有玩家入口，也不会真正扣钱或改变宠物。P1.2c-3 才实现原子扣款、完整重置、永久计数、审计和模糊提交恢复。

## StoneAge 8.0 基线与 Beastbound 差异

本地稳定参考 `/Users/fander/projects/_local_references/StoneAge/gmsv/src/npc/npc_transmigration.c` 保留了成熟的服务端条件检查、宠物选择、损失确认和服务端转生执行方式；没有找到可直接照搬的“所有宠物按形态付费洗转生 + 绑定/非绑定钱包 + GM 动态价格”合同。宠物仓库里的“洗宠”仍是异常/重复宠物整理，不是本功能。

因此 Beastbound 沿用石器的服务端权威与明确确认心智，但价格目录属于私服式长期运营的原创扩展。为了避免过度设计，仍只使用石币和钻石，不新增洗宠券或第五种货币，也不制作独立玩家系统页。

后三个档位目前是预留合同，不代表运行时已经存在商业、进化或融合形态。它们提前给后续宠物设计一个统一比较口径；正式新增形态时仍需根据获取成本、强度和替代难度决定是否沿用。

价格以“重新培养这个形态的机会成本”分层，而不是按个体极品程度收费。这样同一只普通 C 宠的大R可以反复追求完美，但服务器不会因为它已经是 S、洗过很多次或充值较多而暗中提价。教学宠价格较低但不是零，普通宠承担长期石币回收，稀有/奖励宠则用更高机会成本保护其获取价值。

目录也显式覆盖通常只作为 MM、试炼敌人或系统形态出现的行。这不赋予玩家获取或重置资格，只保证未来某只合法实例进入玩家档案时不会走静默兜底；P1.2c-3 仍必须验证宠物真实归属、形态、转生状态和档案完整性。

## GM 覆盖与失败关闭

新增命令 `gm_pet_paid_reset_config` 与 HTTP：

- `GET /gm/pets/paid-reset/config`：返回默认合同、当前覆盖和 32 个形态的最终报价；
- `PUT /gm/pets/paid-reset/config`：整份替换档位覆盖和形态覆盖；
- 更新必须提交 `expectedRevision`，陈旧 revision 明确冲突；
- 形态覆盖优先于档位覆盖，未覆盖项继续使用版本化默认目录；
- 覆盖值严格限制为已知货币、正整数价格和已知钱包策略；未知形态、未知档位、未知字段、损坏持久配置全部失败关闭；
- 每次读取、成功更新或失败尝试都写 GM 审计；成功更新的审计保留 revision 及本次完整档位/形态覆盖值，配置操作不改任何玩家 profile；
- HTTP 方法进入统一 durable 队列，预提交写失败会恢复发布缓存并返回失败，不会让 GM 看到“已成功但重启丢失”的价格。

GM 可以调整运营数值，但不能通过配置改成随次数涨价、只给商业宠开放、保留转生次数、重置后继续绑定或成功后返款。若未来确实要改变这些规则，必须作为新的产品/迁移版本评审，而不是藏在价格 JSON 中。

## 支付计划地基

`pet-paid-reset-payment.js` 只生成纯扣款计划，不修改档案：

- `bound_first_split` 按绑定余额、非绑定余额顺序拆分；
- `unbound_only` 完全忽略绑定余额；
- 总余额不足时返回缺口且 `debits=[]`；
- 每一笔计划写清字段、绑定属性、扣前、扣额和扣后，供 P1.2c-3 在同一权威事务中应用。

本阶段故意不提供“先扣一种、失败再返还”的补偿式写法。下一阶段会把扣款、宠物重置、解绑、永久计数和审计放入同一候选并只在 MySQL COMMIT 后发布。

## 持久化与兼容

- `normalizeData` 新增根字段 `petPaidResetConfig`；旧根缺失时为 `{}`，由严格目录解释成 revision 0。
- MySQL 继续复用 `server_state/auth` 运维文档，不新增全表、不重写 profile、不触碰真实玩家数据库。
- entity-table loader、空实体状态、root contract 和批量 profile 迁移覆盖均包含该字段，防止迁移/回滚漏掉 GM 配置。
- 配置损坏不会自动恢复成默认价或被一次更新覆盖；服务端拒绝报价/更新，等待明确运维修复，避免静默低价洗高价值宠。
- 本阶段不改变 profile schema、宠物实例 schema、协议版本或任何旧宠数值。
