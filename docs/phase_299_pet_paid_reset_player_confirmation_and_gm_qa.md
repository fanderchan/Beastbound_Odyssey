# Phase 299：宠物付费重置玩家确认与 GM 验收入口

## 结论

P1.2c-4 已把 Phase 297 的全形态价格目录和 Phase 298 的原子重置事务接成可实际操作的玩家/GM 闭环：

- 普通玩家只在联网宠物「成长」页、选中合法 authority-v1 一转或二转宠时看到“重置转生”；客户端向服务器读取当前形态价格、绑定/非绑定扣款拆分和固定损失/保留项，不读取隐藏成长或私有审计；
- 第一次点击只进入确认态，不发送请求；第二次按钮明确显示本次货币与金额后才提交。提交携带刚看到的 profile/config revision 和稳定 operationId，陈旧报价只刷新，不自动执行危险动作；
- 成功后使用服务器返回档案刷新为 Lv1/0转/解绑；网络或 COMMIT 结果不明时保留原 operationId，不能生成新 key 盲重试；
- GM/QA 新增“一键准备重置验收档与审计”：首次为当前受权 GM 准备固定一转、二转四灵幼兽和四种验收钱包；以后只读当前价格、永久次数和最近十条安全审计，不重抽、不覆盖、不补发已删除样本；
- legacy/损坏成长档只用临时负样本证明服务器拒绝，不把损坏宠物写进 `auth1373`；玩家响应和 UI 不含 private seed/roll、operationId、原始审计、MySQL 或内部错误码；
- 玩家入口复用现有成长页，没有新增独立系统页、说明书式弹窗或素材需求。Phase 298 的事务说明图继续只是非游戏测试报告，不能称为实机 UI。

## StoneAge 8.0 基线与 Beastbound 差异

本地参考 `/Users/fander/projects/_local_references/StoneAge/gmsv/src/npc/npc_transmigration.c` 的成熟心智是：服务器重新核对目标宠物和转生条件，玩家经过显式选择/确认后才执行。Beastbound 沿用“服务端复核 + 明确确认”，没有把一个危险单击按钮放进宠物页。

Beastbound 的产品决定是所有合法宠物都可按形态付费完整回到 Lv1/0转，并有绑定/非绑定钱包、动态 GM 调价、永久次数和 COMMIT 后成功门槛。这些不是石器 8.0 可直接照搬的合同，因此增加安全公开报价和 operationId；但为了避免过度设计，仍不新增洗宠券、第五种货币、独立商城页、玩家可见审计页或技术恢复按钮。

## 玩家公开报价

`GET /pets/paid-reset/quote?instanceId=...` 是只读权威入口。服务器重新检查：

1. 会话、档案归属、宠物实例和当前 profile revision；
2. 不在战斗或离线挂机，宠物未锁定、未骑乘且不被当前任务占用；
3. 形态存在精确价格行，成长档为合法 authority-v1，培养记录是完整一转或二转；
4. 永久重置次数与私有审计一致；
5. 当前四钱包能否按服务器策略完成扣款。

余额不足仍返回只读报价和差额，让界面可以解释“还差多少”，但提交按钮禁用。其他资格错误失败关闭。公开报价严格只有：

- 宠物实例/形态/名称、当前等级、转数、强化、绑定和永久重置次数；
- profile/config revision；
- 货币、固定金额、可用余额、差额和绑定/非绑定扣款计划；
- 固定结果 Lv1/0转/解绑；
- “清除 / 保留 / 不返还”的稳定枚举。

客户端使用 exact-key 校验；报价多一个私有字段、少一个后果、被篡改扣款或与当前宠物不匹配都会拒绝显示。

## 二次确认与失败恢复

成长页的交互固定为：

1. 初始按钮“重置回 Lv1・0转”；
2. 第一次点击只把按钮改成“再次确认支付 300钻石”等准确文本，并提示立即扣款和不返还；
3. 第二次点击才发送 `POST /pets/paid-reset`；
4. 等待期间按钮锁定为“服务器正在确认……”；
5. 成功应用服务器档案并关闭确认态；profile/config revision 冲突则清报价、重新读取，绝不自动提交；
6. `storage_outcome_unknown` 等结果不明错误保留原 operationId，后续仍用同一操作确认结果；明确业务失败才解除确认并丢弃该 key。

玩家文案只说明游戏结果。COMMIT、receipt、revision 数值、审计 ID 和技术错误码都不进入正常宠物面板。

## GM/QA 验收档

共享 `local_qa_full_v2` 仍只允许本地 QA 账号 `auth1373`，当前精确目录为 server `11`、client `30`、server-authoritative client `8` 项；新入口复用既有 `gm_pet_paid_reset_config` 命令授权，没有额外扩大角色权限。

`POST /gm/pets/paid-reset/qa` 强制幂等 key。首次成功：

- 用正式新宠 factory、正式升级结算和正式转生周期创建 `Lv80/1转`、`Lv88/2转` 两只固定四灵幼兽；
- 样本具有真实 authority-v1 私有成长，但响应只给安全摘要；
- 绑定钻石、非绑定钻石各至少 1,000，绑定石币、非绑定石币各至少 1,000,000，只补到下限，不减少已有余额；
- 二转样本自动成为客户端后续宠物页选择目标；
- ledger 永久记录样本身份，删除或使用后不会补发、覆盖或重抽。

再次执行只刷新当前形态价格、样本是否仍可重置、永久次数和最近十条安全审计。实际损坏负样本只在内存中构造并立刻校验拒绝，不污染 QA 档案。GM 状态文本不展示私有随机数，普通玩家没有该入口。

## 可选人工验收

若之后要亲自确认真实账号链路：

1. 启动本地后端，登录 `auth1373`，打开 GM/QA；
2. 点击“重置验收档与审计”，确认显示两只样本、四钱包、当前 300 钻石报价和 legacy/损坏拒绝自检通过；
3. 点击“宠物”，进入「成长」，确认二转样本显示 `Lv88・2转 → Lv1・0转`，扣款为绑定 250 + 非绑定 50（实际拆分以当时钱包为准）；
4. 第一次点击确认没有扣款或宠物变化，按钮只变为准确金额确认；
5. 第二次点击会真实扣除 QA 账号游戏币并重置该样本。成功后确认 Lv1/0转/解绑、永久次数 +1，Lv1 4V、隐藏成长、强化和技能仍保留；
6. 回到 GM/QA 再点一次入口，确认不补发样本，最近审计显示同一次重置且钱包不被再次补到消费前数值。

自动证据已覆盖合同和界面，这轮人工验收不是合并门槛；只有要判断真实网络等待手感时才需要执行第 4–5 步。
