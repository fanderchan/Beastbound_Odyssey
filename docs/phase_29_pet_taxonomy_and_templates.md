# Phase29：宠物种系 / 亚种 / 形态设计文档

## 目标

Phase29 先不改 Godot 运行逻辑，只把宠物底层分类方式写清楚，作为后续宠物数据表、宠物设计器、捕捉入栏、成长计算和战斗技能继承的基础。

本阶段先设计两个一直用于测试的种系：

- 布伊系
- 乌力系

## 核心结论

宠物采用四层结构：

```text
种系 -> 亚种 -> 形态 -> 属性配比
```

示例：

```text
种系：布伊系
亚种：普通布伊
形态：红色普通布伊
属性配比：10火
```

“亚亚种”这个词不使用，统一改为 **形态**。

原因：

- “形态”更像玩家能理解的外观和成长分支，例如红色、黄色、厚皮、高速。
- 它不会让分类名字越来越绕。
- 后续做宠物图鉴、宠物详情、捕捉预览时，文案会更自然。

## 术语定义

### 种系

`种系` 是宠物的大类，例如 `布伊系`、`乌力系`。

种系主要负责：

- 种族定位。
- 唯一种系被动技能。
- 后续图鉴分类。
- 未来服务器或数据库里的长期宠物身份归类。

字段建议使用：

- `lineId`
- `lineName`
- `passiveSkillId`

说明：

- 一个种系只能绑定一个 `passiveSkillId`。
- 不再设计“被动池”或“形态额外被动”。
- 形态不能换被动，只能通过属性配比、成长倾向和数值，改变这个种系被动的实际效果。
- 运行时为了兼容现有战斗字段，可以继续把最终被动写成 `passiveSkillIds: [line.passiveSkillId]`，但源数据仍然必须是一族一个被动。

### 亚种

`亚种` 是种系下面的战斗分支，例如 `普通布伊`、`普通乌力`。

亚种主要负责：

- 主动技能组。
- 战斗定位。
- 后续是否拥有不同骨架或不同动作包。

字段建议使用：

- `subtypeId`
- `subtypeName`
- `lineId`
- `activeSkillIds`
- `battleRole`

说明：

- 主动技能默认从亚种继承。
- 同一个亚种下面的不同形态，默认拥有同一套主动技能。
- 如果未来某个形态一定要有独立主动技能，必须先单独拍板，不在 Phase29 默认开放。

### 形态

`形态` 是亚种下面的具体宠物模板，例如 `红色普通布伊`、`黄色普通布伊`、`高防乌力`。

形态主要负责：

- 玩家看到的具体宠物名称。
- 颜色或外观差异。
- 属性配比。
- 成长倾向。
- 捕捉难度和生态出现位置。

字段建议使用：

- `formId`
- `formName`
- `lineId`
- `subtypeId`
- `elements`
- `growthProfileId`
- `baseStats`
- `capture`
- `visual`

说明：

- 形态不直接追加被动。
- 形态的属性配比会参与种系被动计算。
- 普通形态不应该把所有能力都写死在自己身上，否则后续会很难维护。

### 属性配比

属性配比用火、水、地、风四个值表示，合计必须为 10。

示例：

```json
{
  "fire": 10,
  "water": 0,
  "earth": 0,
  "wind": 0
}
```

玩家展示时写作：

```text
10火
```

第一版成长倾向：

- `10火`：攻击成长高。
- `10风`：敏捷成长高。
- `10地`：防御成长高。
- `10水`：生命或续航成长高，后续再拍板具体公式。
- 混合属性：后续宠物系统稳定后再设计，不在 Phase29 展开。

## 继承规则

最终进入战斗的宠物能力，按下面顺序合成：

```text
种系唯一被动
+ 亚种主动技能
+ 形态属性和成长
= 战斗宠物模板
```

规则：

- 被动技能只能来自种系。
- 一个种系只有一个种系被动。
- 形态不能追加、替换或随机获得别的被动。
- 主动技能主要来自亚种。
- 形态默认不改主动技能。
- 属性配比和成长倾向来自形态，并且可以作为种系被动的计算输入。
- 战斗中的 `passiveSkillIds` 最终应由 `line.passiveSkillId` 合成得到，而不是在战斗测试脚本里临时写死。

## 种系被动设计规则

种系被动是宠物系统最顶层的身份特征。

设计规则：

- 每个种系只能有一个种系被动。
- 种系被动可以很简单，也可以很复杂。
- 复杂被动应该通过公式读取 `elements`、`baseStats` 或战斗状态，而不是让形态额外塞第二个被动。
- 捕捉后不会随机被动。捕捉到什么形态，就固定拥有这个种系被动和该形态的属性配比。

数据结构建议：

```json
{
  "id": "bui_resistant_skin",
  "label": "抗性皮肤",
  "description": "[被动技能] 根据地水火风属性分别获得对应异常抗性。",
  "effect": {
    "type": "element_scaled_status_resist",
    "scalePerPoint": 0.01,
    "mapping": {
      "earth": "stone",
      "water": "poison",
      "fire": "confusion",
      "wind": "sleep"
    }
  }
}
```

这个结构表达的是：

| 属性 | 对应效果 |
| --- | --- |
| 地 | 石化抗性 |
| 水 | 中毒抗性 |
| 火 | 混乱抗性 |
| 风 | 睡眠抗性 |

计算方式：

```text
对应抗性 = 属性点数 * 1%
```

例如：

- `10火`：10% 混乱抗性。
- `10风`：10% 睡眠抗性。
- `10地`：10% 石化抗性。
- `10水`：10% 中毒抗性。
- `地9风1`：9% 石化抗性 + 1% 睡眠抗性。

这只是第一版设计值。后续如果觉得 1% 太低或太高，可以只改 `scalePerPoint`，不需要重做宠物分类。

## 布伊系第一版

### 种系

| 字段 | 值 |
| --- | --- |
| `lineId` | `bui` |
| 种系名 | 布伊系 |
| 定位 | 早期伙伴型宠物，适合做新手测试宠 |
| 种系被动 | 抗性皮肤 |

说明：

- 所有布伊系都拥有 `抗性皮肤`。
- 不同布伊形态不换被动，只通过属性配比改变 `抗性皮肤` 的实际抗性。
- 这样可以表达复杂被动，也不会让“红色布伊一个被动、黄色布伊另一个被动”变得混乱。

### 被动：抗性皮肤

玩家文案：

```text
抗性皮肤：[被动技能] 根据地水火风属性分别获得对应异常抗性。
```

第一版规则：

| 属性 | 抗性 |
| --- | --- |
| 地 | 石化抗性 |
| 水 | 中毒抗性 |
| 火 | 混乱抗性 |
| 风 | 睡眠抗性 |

计算：

```text
每 1 点属性 = 1% 对应异常抗性
```

### 亚种：普通布伊

| 字段 | 值 |
| --- | --- |
| `subtypeId` | `bui_normal` |
| 亚种名 | 普通布伊 |
| 所属种系 | 布伊系 |
| 战斗定位 | 平衡型早期宠物 |
| 主动技能 | 攻击、防御、布伊冲撞 |

主动技能 ID：

```text
pet_attack
pet_defend
pet_bui_charge
```

### 形态

| `formId` | 形态名 | 属性配比 | 成长倾向 | 抗性皮肤实际效果 | 用途 |
| --- | --- | --- | --- | --- | --- |
| `bui_normal_red_fire10` | 红色普通布伊 | 10火 | 攻击成长高 | 10% 混乱抗性 | 主人受控宠物候选 |
| `bui_normal_yellow_wind10` | 黄色普通布伊 | 10风 | 敏捷成长高 | 10% 睡眠抗性 | 高速布伊测试 |
| `bui_normal_thick_earth10` | 厚皮布伊 | 10地 | 防御成长高 | 10% 石化抗性 | 现有异常状态测试 |

## 乌力系第一版

### 种系

| 字段 | 值 |
| --- | --- |
| `lineId` | `wuli` |
| 种系名 | 乌力系 |
| 定位 | 常见野生兽类，适合测试敌方、捕捉、状态抗性 |
| 种系被动 | 硬壳体质 |

说明：

- 所有乌力系都拥有 `硬壳体质`。
- 不同乌力形态不换被动，只通过属性配比改变 `硬壳体质` 的实际效果。
- `高防乌力` 通过 10地 把石化抗性推到 100%，表现为石化免疫。
- `高速乌力` 的高速来自 10风 成长倾向，不再额外挂 `迅捷本能` 被动。

### 被动：硬壳体质

玩家文案：

```text
硬壳体质：[被动技能] 根据地属性获得石化抗性；地属性达到10时免疫石化。
```

第一版规则：

```text
石化抗性 = 地属性点数 * 10%
```

例如：

- `10地`：100% 石化抗性，等同石化免疫。
- `5地5风`：50% 石化抗性，同时敏捷成长也会受到风属性影响。
- `10风`：0% 石化抗性，但敏捷成长高。

### 亚种：普通乌力

| 字段 | 值 |
| --- | --- |
| `subtypeId` | `wuli_normal` |
| 亚种名 | 普通乌力 |
| 所属种系 | 乌力系 |
| 战斗定位 | 野生敌方测试宠 |
| 主动技能 | 攻击、防御 |

主动技能 ID：

```text
pet_attack
pet_defend
```

说明：

- Phase29 不急着给乌力系塞很多主动技能。
- 等宠物主动技能设计开始后，再决定乌力系是否拥有吼叫、冲撞、状态技能等专属分支。

### 形态

| `formId` | 形态名 | 属性配比 | 成长倾向 | 硬壳体质实际效果 | 用途 |
| --- | --- | --- | --- | --- | --- |
| `wuli_normal_orange_fire10` | 普通乌力 | 10火 | 攻击成长高 | 0% 石化抗性 | 当前普通敌人 |
| `wuli_normal_fast_wind10` | 高速乌力 | 10风 | 敏捷成长高 | 0% 石化抗性 | 行动顺序测试 |
| `wuli_normal_tough_earth10` | 高防乌力 | 10地 | 防御成长高 | 100% 石化抗性 | 状态免疫测试 |

## 和当前战斗系统的关系

Phase28 已经有：

- `battle_passive_skills.json`
- `passiveSkillIds`
- 被动悬停显示
- 石化免疫、毒素抗性、迅捷本能

Phase29 把被动来源改成更稳定的模型：

- 宠物模板源数据里，一个种系只写一个 `passiveSkillId`。
- 运行时可以继续用现有 `passiveSkillIds` 数组承接结果。
- `抗性皮肤`、`硬壳体质` 这类被动需要读取形态的 `elements` 后再计算具体效果。

后续真正落地时：

- `高防乌力` 不再额外挂 `stone_immunity`，而是通过乌力系 `硬壳体质` + `10地` 计算出 100% 石化抗性。
- `厚皮布伊` 不再额外挂离散被动，而是通过布伊系 `抗性皮肤` + `10地` 计算出 10% 石化抗性。
- `黄色普通布伊` 不再额外挂 `quick_instinct`，而是通过 `10风` 走敏捷成长，并通过 `抗性皮肤` 获得 10% 睡眠抗性。
- 旧的 `stone_immunity`、`poison_resistance`、`quick_instinct` 可以保留为 Phase30 前的测试占位，但正式宠物模板不再用“形态追加离散被动”的方式设计。

## 不做范围

Phase29 不做：

- 不改 Godot 代码。
- 不新增 `pet_templates.json`。
- 不改现有战斗测试场景。
- 不做宠物设计器 UI。
- 不做正式成长公式。
- 不做捕捉后入栏。
- 不做后端数据库表。
- 不接正式宠物美术。

这些等你确认文档方向后，再进入下一阶段。

## 后续阶段建议

Phase30 可以做宠物模板数据落地：

- 新增 `client/godot/data/pet_templates.json`。
- 新增宠物模板加载与校验脚本。
- 把现有布伊、乌力测试单位改为从宠物模板合成。
- 自动校验属性配比总和必须为 10。
- 自动校验每个种系必须且只能有一个 `passiveSkillId`。
- 自动校验主动技能 ID 和被动技能 ID 都能在现有目录里找到。
- 支持 `element_scaled_status_resist` 这类读取属性配比的参数化被动。

Phase31 可以做宠物详情 / 图鉴雏形：

- 鼠标或触屏查看宠物时显示种系、亚种、形态、属性配比、被动技能。
- 玩家可见信息保持简洁，不显示调试字段。

## 人工验收点

请重点确认：

- `形态` 这个名字是否比 `亚亚种` 更舒服。
- `种系 -> 亚种 -> 形态 -> 属性配比` 是否符合你后续设计大量宠物的习惯。
- 布伊系和乌力系第一版是否符合当前测试需要。
- `黄色普通布伊：10风，敏捷成长高` 是否符合你的设定。
- 高防乌力通过形态追加石化免疫，这个归属是否合理。
