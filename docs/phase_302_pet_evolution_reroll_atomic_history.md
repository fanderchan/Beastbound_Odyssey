# Phase 302：宠物进化二代重抽、原子事务与成长履历

## 结论

P1.3c 已按用户最终决定落地为“重抽结果，保留公开历史”：

- 源宠必须是指定形态、恰好一转、Lv140，并达到同形态一转 Lv140 成长战力的前 10%；
- 成功后仍是同一个宠物实例，但切换为进化形态并回到 Lv1；
- 目标二代宠的 Lv1 血/攻/防/敏和隐藏每级成长都按目标物种重新生成，不继承源宠逐属性分位；
- 一转培养加成、强化、技能、名字、归属、绑定/锁定和培养审计继续保留；
- 源宠 0转和1转的玩家可见成长证据写入进化履历，私有 seed/roll 不保存到公开履历，也不会返回客户端；
- 宠物成长页沿用原来的三个阶段页签，第三页统一为“2转/进化/融合”。进化宠的“1转成长”页展示进化前原宠的 Lv1 初始值与 Lv140 实绩，第三页展示重新抽取后的当前进化宠。

生产目录仍保持 `runtimeEnabled=false`，两只进化宠的正式 2.5D 资产未完成前，玩家无法执行真实进化，也不会扣除材料或石币。

## StoneAge 8.0 基线与 Beastbound 取舍

StoneAge 8.0 的成熟基线仍然是高阶宠物转换由服务端复核、进化/融合与普通转生互斥。Beastbound 保留这两点，但“进化后二代重新抽品质，并允许玩家回看源宠成长履历”是本项目自己的产品规则。

这不是额外增加一个进化说明系统：成长履历直接复用现有 0转/1转/2转页签。它解决的具体问题是，玩家投入一只一转高水平源宠后，仍能学习“怎样的源宠最终换来了这只进化宠”，同时保留重抽带来的惊喜。客户端只展示玩家本来能够通过练级看到的公开数值，不展示隐藏成长分位或私有随机身份。

## 一转 P90 入场门槛

门槛使用不含装备、当前血量和外部临时增益的成长战力：

`round(Lv140生命 × 0.25 + Lv140攻击 + Lv140防御 + Lv140敏捷)`

离线审计对每个源形态分别生成 10,000 只一转 Lv140 样本，固定同形态 P90 阈值：

| 路线 | 源形态 | P90 成长战力 | 20,000 样本复核放行率 |
| --- | --- | ---: | ---: |
| 晶甲乌力 | 高防乌力 | 1345 | 10.43% |
| 月岚风狐 | 高地风狐 | 1437 | 10.32% |

该门槛只决定源宠是否有资格进化，不预测或影响进化后二代的重抽结果。玩家可以在成长页直接看到进化前一转 Lv140 实绩，但不会看到隐藏成长 seed、roll 或未来结果。

## 二代生成与公开履历

服务端进化时先严格复核源宠，再创建目标形态的新 Lv1 成长身份：

1. 复制同实例的可保留身份、培养、强化和技能字段；
2. 清除源形态的私有成长、公开成长观察、等级、经验和临时战斗字段；
3. 通过统一新宠工厂按目标物种重新生成 Lv1 4V 与隐藏成长；
4. 恢复一转培养加成，使目标宠仍属于一转后的进化终局分支；
5. 写入只含公开证据的 `evolutionLineage.stageSnapshots`：0转重建实绩和1转实际实绩；
6. 最终校验实例 ID 不变、形态已切换、等级为 Lv1、源/目标私有随机身份不同。

两条路线各 10,000 次独立重抽审计中，源宠与目标宠各属性最大相关系数分别为 `0.017094` 与 `0.015607`，接近独立随机；重抽后的成长增量中位数分别为 `1.806292` 与 `1.818178`，落在普通二转第二阶段批准的中心预算带。由于源宠和目标宠独立抽取，单只宠物之间的增量区间会有意拉宽，这正是重抽惊喜，不再要求每个样本逐分位贴合旧宠。系统既没有把旧分位偷偷映射到新宠，也没有让目标物种的中心成长预算失控。

公开资料投影对进化履历实行白名单：只允许阶段、形态、Lv1 4V、Lv140 实绩、成长观察与成长战力。测试向履历各层注入 `privateSeed/privateRoll` 后，玩家响应中均不存在这些字段。

## 服务端原子事务

玩家入口分为只读报价和确认执行：

- `GET /pets/evolution/quote` 只检查当前形态、1转/Lv140/P90、资格、正式资产、材料与绑定/非绑定石币，不修改档案；
- `POST /pets/evolution` 必须携带 operation ID、当前 profile revision 和目录 ID；规则或档案变化会要求刷新后重新确认；
- 同一持久事务候选同时包含宠物重抽、材料扣除、绑定优先石币扣除、公开履历和操作回执；
- MySQL 真正 COMMIT 前不会向玩家发布成功、扣款或新宠；回滚后源宠、材料和两种石币都保持原样；
- 同一 operation ID 重放只返回已提交结果，不会再次扣除或再次重抽；服务重启后仍能从持久回执恢复同一结果。

失败路径覆盖低于 P90、缺资格、缺任一材料、石币不足、宠物锁定/骑乘、档案 revision 过期、目录版本过期、正式资产未就绪和写入回滚，全部要求零变更。

## 客户端成长履历

`PetGrowthObservationModel.evaluate_pet()` 继续只评价当前宠物，捕捉筛选、排序和人工去留不会被旧历史污染。只有成长页切换阶段时使用 `evaluate_pet_for_stage()`：

- `0转成长`：进化前源宠的 0转公开实绩；
- `1转成长`：进化前源宠的一转 Lv140 实绩，标题明确标注“进化前”；
- `2转/进化/融合`：当前进化后二代宠的 Lv1 4V和后续成长观察。

历史页隐藏当前宠的人工参考线编辑，避免玩家误把源宠历史阈值保存成新宠筛选策略。两张雷达图也分别标注“进化前 Lv1 4V分位”和“进化前1转成长分位”。

真实 Metal 1280×720 证据：`.run/evidence/p1_3c_evolution_history.png`。截图停在“1转成长”，可同时看到“2转/进化/融合”页签、源形态“高防乌力”、Lv140 实绩和进化前双雷达；它是游戏 `Main.tscn` 画面，不是说明图。

## 文件与兼容边界

- `pet_evolution_balance.json`、`pet_evolution_routes.json`：P90、二代重抽、终局、成本和历史合同；
- `pet-evolution.js`：纯资格检查、二代生成、履历与结果验证；
- `pet-evolution-domain.js`：账号、保护状态、资格、资产和原子档案变更；
- `auth-service.js`、`durable-mutation-state.js`、`http-server.js`：只读报价、持久执行、COMMIT/重放和 HTTP 边界；
- `profile-visibility.js`：进化履历公开白名单；
- `pet_growth_observation_model.gd`、`panel_flow_coordinator.gd`：三阶段成长显示；
- `pet_evolution_eligibility_audit.mjs`、`pet_evolution_route_audit.mjs`：P90 与独立重抽分布门禁。

本项只作用于未来明确确认的进化，不迁移或重写任何旧宠物。测试使用 memory/隔离 HTTP/隔离持久存储，没有连接共享 MySQL、修改 `auth1373` 或操作真实玩家数据。
