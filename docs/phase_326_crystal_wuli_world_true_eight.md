## 原创身份与真八向生产

本次目标是高防乌力 `wuli_normal_tough_earth10` 的进化形态 `wuli_evolved_crystal_earth8_water2`，没有把普通橙色乌力当作进化源。晶甲乌力以低重心野猪体型、巨型额盾、双肩晶堡、分层背甲、青蓝裂隙、蓝眼和尾端晶芽作为跨方向身份锁；整体骑乘使用成年见习猎人，并要求人物头身、坐姿和运动轴与坐骑一致。

生产按 `south / southwest / west / northwest / north / northeast / east / southeast` 八个方向分别生成。宠物首批第二组的东、东南两行实际读成西、西南，10 张候选当场拒绝并只保留在来源归档中，没有安装进运行目录；随后分别以专用真东、真东南提示重新生成各 `idle 1 + walk 4`。最终宠物来源归档为 50 张，其中 40 张入选、10 张拒绝；整体骑乘来源归档与入选均为 40 张。

这次退回证明“提示写了东”不能充当方向通过证据。安装门槛继续以实际轮廓为准：正面/背面轴、口鼻与尾部位置、背甲可见面积，以及骑手头身和坐骑轴都必须能与相邻方向明确区分。

## 色键缩放残边修复

批量管线原先在洋红色键抠图后使用 premultiplied Lanczos 缩放。复现证明，输入可见像素中即使已经没有接近色键的颜色，Lanczos 的负瓣仍可能由合法边缘颜色产生洋红状过冲。最初尝试的“依据整帧色键来源，在缩放后按 RGB/色距清理”被提交前独立审查否决：到缩放阶段已经失去逐像素资格证明，这种做法可能误删作者绘制的半透明紫色，因此没有提交或作为最终证据使用。

最终方案不在缩放后删除任何颜色：

1. `chroma_key` 输入在裁切到 512px 源帧、再从 512px 派生 256px 运行帧的两次缩放中，alpha 与所有 premultiplied RGB 通道统一使用 BILINEAR；非负权重避免 Lanczos 负瓣过冲，透明背景也不贡献 RGB；
2. 已透明输入与旧 metadata 缺失模式标记的 bundle 继续使用原 premultiplied Lanczos，保持旧资产字节兼容；
3. 两条路径只保留全局既有的 `alpha < 2` 归零，不执行任何缩放后 RGB、色距或“近洋红”自动清理；
4. 可见色键状残边只触发 QC 失败，绝不猜测其来源并改写像素；
5. 构建器逐帧记录 `sourceResampleMode` 与 `runtimeResampleMode`。安装器要求新格式的两字段同时存在、为受限枚举、彼此相等，并强制 `chroma_key` 对应 BILINEAR、`transparent_alpha` 对应 Lanczos；只有两字段都不存在时才按旧包兼容为 Lanczos，仅出现其中一个字段、伪造或不匹配均失败关闭；
6. 回归直接覆盖半透明紫色/接近色键的作者细节在 512px 与 256px 两段仍可见且保色、可见洋红只能使构建失败、旧缺标记包继续按 Lanczos 重放，以及安装器逐字节一致。

构建器 `30/30`、安装器 `18/18`，合计 `48/48` 通过。六组共 90 张候选帧全部重建，源帧与运行帧的可见洋红残边均为 `0`，每帧均记录成对 `premultiplied_bilinear`；其中 `pet-b` 的错误东/东南 10 张只保留在来源归档。最终安装的宠物与整体骑乘 80 张运行帧以及 90 张 512px 来源帧，`alpha=1` 与透明区 RGB 泄漏均为 `0`。来源 ledger 同时绑定入选/拒绝源帧、运行帧路径、文件 SHA-256、解码 RGBA SHA-256 和明确缩放模式。
